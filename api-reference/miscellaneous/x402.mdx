title: "Accountless x402 API Payments"
description: "Call supported NanoGPT API endpoints without an account or API key, receive a 402 payment quote, pay with a supported crypto rail, and complete the original request."
---

## Overview

NanoGPT supports accountless API payments for selected endpoints. Send a normal API request without `Authorization` or `x-api-key`. If the endpoint supports accountless payment and the request can be quoted, NanoGPT returns `402 Payment Required` with payment options.

After paying, complete the request in one of two ways:

- For polling schemes such as Nano or Base USDC address payments, call the `completeUrl` from the quote.
- For exact x402 schemes, replay the same request with an `X-PAYMENT` header.

Only endpoints returned by `GET /api/v1/x402/endpoints` should be treated as public accountless x402 endpoints.

## Endpoint Matrix

Use the endpoint matrix as the source of truth for the current deployment:

```bash
curl https://nano-gpt.com/api/v1/x402/endpoints
```

To include deferred and unsupported entries:

```bash
curl "https://nano-gpt.com/api/v1/x402/endpoints?supported=all"
```

When x402 is disabled, payment storage is unavailable, or completion replay is not configured, the default supported list can be empty.

Current public accountless endpoints:

| Method | Endpoint | Streaming | Async | Quote type | Notes |
| --- | --- | --- | --- | --- | --- |
| `POST` | `/api/v1/images/generations` | unsupported | no | deterministic | Recommended first public milestone. |
| `POST` | `/api/v1/images/edits` | unsupported | no | deterministic | JSON edit requests only. Multipart uploads require normal auth before conversion. |
| `POST` | `/api/v1/chat/completions` | beta | no | estimated with reconciliation | Non-streaming is stable. Streaming is beta. |
| `POST` | `/api/v1/responses` | beta | no | estimated with reconciliation | Non-streaming is stable. Streaming and background mode are beta. |
| `POST` | `/api/v1/data/web/search` | unsupported | no | deterministic | Public v1 data path for web search. |
| `POST` | `/api/v1/data/url/scrape` | unsupported | no | deterministic | Public v1 data path for URL scraping. Use `urls: [...]`. |

Do not assume other API endpoints support accountless payment until they appear as supported in the matrix and have passed live smoke tests.

## Payment Schemes

The `payment.accepted[]` array and endpoint matrix show which schemes are enabled in the running deployment.

| Public scheme | Network | Asset | Flow |
| --- | --- | --- | --- |
| `nano` | `nano-mainnet` | XNO | Send to an ephemeral Nano address, then call `completeUrl`. |
| `nano-exact` | Nano mainnet exact settlement | XNO | Sign an exact payment payload and replay the original request with `X-PAYMENT`. |
| `x402-exact` | Base | USDC | EIP-3009 exact payment in `X-PAYMENT`, then replay the original request. |
| `base-usdc` | Base | USDC | Send to an ephemeral Base address, then call `completeUrl`. |

Lightning/L402 is not currently documented as a NanoGPT-supported accountless payment rail.

## 402 Response Shape

For accountless payment challenges, key off the stable top-level `payment` object. The `error` object is kept for OpenAI-compatible error handling and may vary by endpoint. The legacy `accepts`, `x402Version`, and top-level `requestHash` fields may still appear for backwards compatibility and advanced protocol clients.

```json
{
  "error": {
    "type": "insufficient_quota",
    "code": "insufficient_quota",
    "message": "Payment required to complete this request."
  },
  "payment": {
    "version": 1,
    "paymentId": "pay_...",
    "requestHash": "sha256:...",
    "expiresAt": "2026-06-02T16:30:00.000Z",
    "amountUsd": "0.0714",
    "statusUrl": "https://nano-gpt.com/api/x402/status/pay_...",
    "completeUrl": "https://nano-gpt.com/api/x402/complete/pay_...",
    "accepted": [
      {
        "scheme": "nano",
        "protocolScheme": "nano",
        "network": "nano-mainnet",
        "amount": "...",
        "amountFormatted": "0.17067988 XNO",
        "amountUsd": "0.0714",
        "payTo": "nano_...",
        "paymentId": "pay_...",
        "statusUrl": "https://nano-gpt.com/api/x402/status/pay_...",
        "completeUrl": "https://nano-gpt.com/api/x402/complete/pay_..."
      }
    ]
  },
  "x402Version": 1,
  "accepts": []
}
```

Field guidance:

- `payment.paymentId`: stable ID for status and completion.
- `payment.requestHash`: SHA-256 binding for the original method, path, and body.
- `payment.expiresAt`: ISO timestamp for payment expiry.
- `payment.amountUsd`: quoted request cost as a USD string.
- `payment.statusUrl`: primary option status URL when the primary option supports status polling.
- `payment.completeUrl`: primary option completion URL when the primary option supports `completeUrl`.
- `payment.accepted[]`: normalized public payment options.
- `payment.accepted[].statusUrl` and `payment.accepted[].completeUrl`: scheme-specific URLs. Prefer these after selecting a payment scheme.
- `accepts[]`: lower-level protocol requirements kept for compatibility and advanced clients.

Base `x402-exact` replay options should not be expected to include a status URL unless the implementation starts updating the original quote record after replay.

## Example: Image Generation With Nano Complete Flow

Send the normal request without an API key:

```bash
curl -i https://nano-gpt.com/api/v1/images/generations \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-image-1",
    "prompt": "a product photo of a brushed steel desk lamp",
    "size": "1024x1024",
    "n": 1
  }'
```

Expected response: `402 Payment Required` with `payment.accepted[]`.

After sending the exact XNO amount to the selected `nano` option's `payTo` address, call its `completeUrl`:

```bash
curl -i -X POST "https://nano-gpt.com/api/x402/complete/pay_..."
```

The complete response is the original image generation response. NanoGPT replays the stored original request, so the completed request is bound to the quoted method, path, body hash, amount, and expiration.

## Example: Exact Replay Flow

Exact schemes use `X-PAYMENT`. Replay the same request that produced the quote:

```bash
curl -i https://nano-gpt.com/api/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-PAYMENT: $BASE64_PAYMENT_PAYLOAD" \
  -d '{
    "model": "gpt-4.1-nano",
    "messages": [{"role": "user", "content": "Say ok"}]
  }'
```

NanoGPT validates exact `X-PAYMENT` replay against the quoted resource, amount, and expiration. Clients should replay the same request they quoted unless endpoint docs state otherwise. Successful exact settlement may include `X-PAYMENT-RESPONSE`.

## Status Polling

Polling applies to quote-and-complete rails such as `nano` and `base-usdc`.

```http
GET /api/x402/status/{paymentId}
POST /api/x402/complete/{paymentId}
```

Known statuses:

- `pending`
- `underpaid`
- `paid`
- `processing`
- `completed`
- `failed`
- `refunded`
- `expired`

Status responses include `pollAfterSeconds`, plus `Retry-After` and `X-Poll-After` headers.

Async media status URLs are not documented as stable accountless x402 behavior until signed unauthenticated status and media URL behavior is smoke-tested and expiration is explicitly documented.

## Pricing And Reconciliation

Image and data endpoint quotes are deterministic for the submitted request parameters.

Chat Completions and Responses quotes are estimates and are reconciled after provider usage is known. If the final provider cost is lower, endpoint-specific reconciliation may refund or credit according to the implementation path. If provider execution fails after payment, endpoint-specific failed-fee reconciliation or refund behavior applies.

Do not assume every accountless x402 payment is an exact final charge.

## Security Notes

- `completeUrl` payments are bound to the original request method, path, body hash, quote amount, and expiration.
- Exact `X-PAYMENT` replay is currently resource, amount, and expiration bound. Do not assume full body-hash binding unless the endpoint explicitly documents it.
- Expired payments cannot be completed.
- Underpaid payments cannot be completed.
- Completed or processing payments cannot be completed twice.
- Payment credentials must not be logged by clients.
- Do not send secrets in unauthenticated x402 requests unless you are comfortable with the endpoint's normal request handling and retention policy.
