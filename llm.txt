Title: NanoGPT API LLM Guide
Audience: coding assistants helping devs integrate NanoGPT
Last updated: 2026-06-04

GOAL
Equip developers to call the NanoGPT API for chat, batch jobs, streaming, memory, search, embeddings, and audio with OpenAI-compatible clients in minutes.

STYLE FOR ANSWERS
- Prefer TypeScript (Node 18+) with openai@^5.8.0 and baseURL set to https://nano-gpt.com/api.
- Share Python or curl snippets when requested or when automation beyond Node is the better fit.
- Keep examples runnable, scoped to one task, and load secrets from environment variables.
- Reference GET /api/v1/models for model availability instead of guessing ids or capabilities.
- Explain reasoning fields, streaming frames, and billing metadata whenever they affect integration.
- If unsure, acknowledge it and point the developer to LINKS.

REQUIRED PACKAGES AND VERSIONS
- npm install openai@^5.8.0
- yarn add openai@^5.8.0 (same SDK for Yarn projects)
- pnpm add openai@^5.8.0 (monorepo-friendly install)
- pip install --upgrade openai
- Node 18+ or Deno/Bun with fetch and Web Streams support

CORE CONCEPTS
- API base https://nano-gpt.com/api hosts OpenAI-style chat/completions/embeddings plus /v1/audio/speech; Anthropic /v1/messages lives at https://nano-gpt.com/v1/messages.
- Batch API base https://api.nano-gpt.com/api/v1 hosts /files and /batches for asynchronous OpenAI-compatible chat completion jobs. Use api.nano-gpt.com for file upload, batch creation, polling, cancellation, and output downloads; do not use nano-gpt.com for batch uploads.
- Authentication uses Authorization: Bearer or x-api-key headers; generate keys in the NanoGPT dashboard.
- Accountless x402: selected endpoints can be called without Authorization or x-api-key to receive 402 Payment Required with a top-level payment quote. Check GET /api/v1/x402/endpoints for the supported matrix; stable public endpoints include non-streaming chat completions, non-streaming responses, image generations, JSON image edits, /api/v1/data/web/search, and /api/v1/data/url/scrape.
- Compatibility: standard OpenAI schema (messages, tools, response_format, seed, reasoning, parallel_tool_calls) is preserved for sync and streaming calls.
- Model suffixes: append :online or :online/linkup-deep for Linkup search, :memory or :memory-<days> for Polychat memory, and combine suffixes as needed.
- Data API: /api/v1/data is the stable discovery and dispatch layer for web search, URL scraping, Firecrawl, Google Maps, Reddit, LinkedIn, Instagram, TikTok, X/Twitter, and Hunter enrichment. It wraps existing direct endpoints without changing their response schemas or billing.
- Provider routing: on /v1/chat/completions, /v1/completions, and /talk-to-gpt, body provider can be a string provider ID or an object with order, only, ignore, sort, max_price, allow_fallbacks, and require_parameters. order is a soft preference; only is a hard pin; unknown providers in only return provider_unknown_provider, while unknown providers in order/ignore are ignored.
- Distillation policy: text model records and provider rows may include distillationPolicy with status, label, basis, sourceUrl, and note. Use /api/explore/text-models?distillation=allowed or /api/explore/search?type=text&q=...&distillation=allowed to filter model-level allowed text models. This is API-only; the text models browser UI does not show a distillation filter or badge.
- Reasoning and pricing: request reasoning via the reasoning object, read delta.reasoning in streams, and handle x_nanogpt_pricing as an extension object on final streaming chunks or non-stream JSON responses.
- Usage API: GET /api/v1/usage returns aggregate spend, refund, request, and token usage for the authenticated API key by UTC day, public model label, or both. It does not expose prompts, responses, raw transaction rows, provider routing, or account-wide usage across multiple keys.
- Caching and memory: most providers use implicit caching automatically when supported; top-level caching: true routes to a cache-capable provider and defaults to stickyprovider: true; prompt_caching/cache_control is for explicit Claude cache controls (5m or 1h TTL), and memory compression replaces prior turns before dispatch.

GLOSSARY
- Linkup search: NanoGPT web search triggered by :online suffix; prepends fresh results and bills $0.006 (standard) or $0.06 (deep).
- Data API endpoint IDs: stable tool IDs such as web.search, url.scrape, google_maps.search, reddit.scrape, x.read, and hunter.domain_search for dynamic clients that choose data tools at runtime.
- Polychat memory: hierarchical compression invoked by :memory suffix or memory header; stores 1-365 day summaries and replaces conversation history.
- Prompt caching: implicit for providers that support it (including OpenAI, Gemini, and many open-source provider routes); top-level caching: true is capability-based provider routing, not cache annotation; explicit controls via prompt_caching/promptCaching/cache_control are mainly for Claude and support optional cutAfterMessageIndex.
- Distillation policy: API-only metadata indicating whether model or provider-route outputs may be used for output-based model training or distillation under NanoGPT's recorded model-license and provider-terms rules. Status values are allowed, disallowed, and unknown.
- Reasoning stream: delta.reasoning text emitted by reasoning-enabled models; reasoning_details provides structured metadata when available.
- x_nanogpt_pricing: billing extension object with stable core fields (amount, currency, sanitized error) plus provider-dependent optional metadata (cost, paymentSource, token counters, team billing flags).
- BYOK: bring-your-own-key mode activated with x-use-byok: true or byok.enabled: true to forward requests using stored provider credentials.
- x402 payment rails: production 402 quotes may advertise nano, nano-exact, x402-exact, and base-usdc. Lightning/L402 is not currently advertised by NanoGPT 402 responses.

HARD RULES AND GOTCHAS
- Always set baseURL to https://nano-gpt.com/api (or https://nano-gpt.com for /v1/messages) and include Authorization or x-api-key on every call.
- Exception to auth: for accountless x402 on supported endpoints, intentionally omit Authorization and x-api-key, key off the top-level payment object in the 402 response, then call completeUrl for nano/base-usdc or replay the same request with X-PAYMENT for exact schemes.
- Do not hardcode model support; call GET /api/v1/models?detailed=true to confirm vision, context length, and subscription status.
- Do not guess provider IDs. Use GET /api/models/:canonicalId/providers for public provider IDs and pricing. Avoid internal provider names that are not returned by public discovery.
- For distillation workflows, inspect distillationPolicy.sourceUrl and distillationPolicy.note on both the model and selected provider route. Provider-specific restrictions override model-level allowances, and unknown model/license status remains unknown when a provider is silent.
- Handle streaming as SSE: group chunks by id, concatenate delta.content, capture delta.reasoning separately, stop on [DONE], and read usage/x_nanogpt_pricing on the final chunk.
- Never rely on <think> tags; reasoning text lives in reasoning/reasoning_details and may be omitted when reasoning.exclude is true.
- Memory replaces the entire history with a compressed summary; send the up-to-date messages each turn and avoid mixing original transcripts with memory output.
- BYOK requests skip fallback providers; warn developers that misconfigured downstream keys return provider errors directly without NanoGPT failover.
- Batch API JSONL rows must use method POST, url /v1/chat/completions, unique custom_id values, body.messages, body.model, and body.max_tokens or body.max_completion_tokens. All rows in one batch must use the same model and model family; do not mix GPT, Claude, and Gemini rows. Streaming, tools/functions, response_format, audio, and non-image multimodal bodies are not supported.

QUICKSTART MINIMAL
Set up the OpenAI SDK once and call the chat completions endpoint with familiar parameters.

```ts
// quickstart.ts
import OpenAI from 'openai';

const client = new OpenAI({
  apiKey: process.env.NANOGPT_API_KEY!,
  baseURL: 'https://nano-gpt.com/api',
});

async function run() {
  const completion = await client.chat.completions.create({
    model: 'google/gemini-3-flash-preview',
    messages: [
      { role: 'system', content: 'You are the NanoGPT onboarding guide.' },
      { role: 'user', content: 'Summarize the API authentication steps.' },
    ],
    reasoning: { enabled: true, effort: 'low' },
  });

  console.log(completion.choices[0]?.message.content);
}

run().catch((err) => {
  console.error('NanoGPT error:', err.response?.data ?? err);
});
```

BATCH API MINIMAL
Use asynchronous batches for high-volume offline chat completion jobs where latency is not important. Supported rows target /v1/chat/completions for supported GPT, Claude, and Gemini models, including text and image_url inputs.

```bash
cat > batch.jsonl <<'JSONL'
{"custom_id":"request-1","method":"POST","url":"/v1/chat/completions","body":{"model":"gpt-4.1-mini","messages":[{"role":"user","content":"Summarize this in one sentence."}],"max_tokens":64}}
JSONL

curl https://api.nano-gpt.com/api/v1/files \
  -H "Authorization: Bearer $NANOGPT_API_KEY" \
  -F purpose=batch \
  -F file=@batch.jsonl

curl https://api.nano-gpt.com/api/v1/batches \
  -H "Authorization: Bearer $NANOGPT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"input_file_id":"file_abc123","endpoint":"/v1/chat/completions","completion_window":"24h"}'
```

STREAMING WITH REASONING
Use stream: true to read delta.content and delta.reasoning while capturing final-chunk pricing metadata.

```ts
import OpenAI from 'openai';

const client = new OpenAI({
  apiKey: process.env.NANOGPT_API_KEY!,
  baseURL: 'https://nano-gpt.com/api',
});

async function streamSummary() {
  const stream = await client.chat.completions.create({
    model: 'openai/gpt-5.2',
    stream: true,
    messages: [
      { role: 'system', content: 'Respond concisely and expose your reasoning.' },
      { role: 'user', content: 'Describe the NanoGPT streaming contract.' },
    ],
    reasoning: { enabled: true, effort: 'medium' },
  });

  let output = '';
  let reasoning = '';
  for await (const chunk of stream) {
    const delta = chunk.choices?.[0]?.delta;
    if (delta?.reasoning) reasoning += delta.reasoning;
    if (delta?.content) output += delta.content;
    const pricing = (chunk as any).x_nanogpt_pricing;
    if (pricing) console.info('final pricing metadata', pricing);
  }

  console.log({ reasoning, output });
}

streamSummary().catch(console.error);
```

TOOLS AND STRUCTURED OUTPUT
NanoGPT forwards OpenAI tool definitions and structured_outputs to the selected provider.

```ts
const result = await client.chat.completions.create({
  model: 'openai/gpt-5.2',
  messages: [
    { role: 'system', content: 'You call tools to fetch weather data.' },
    { role: 'user', content: 'What is the weather in Austin right now?' },
  ],
  tools: [
    {
      type: 'function',
      function: {
        name: 'getWeather',
        description: 'Return current weather conditions.',
        parameters: {
          type: 'object',
          properties: {
            location: { type: 'string', description: 'City, Country' },
          },
          required: ['location'],
        },
      },
    },
  ],
  tool_choice: 'auto',
  parallel_tool_calls: false,
});

const call = result.choices[0]?.message.tool_calls?.[0];
if (call) {
  console.log('tool name', call.function.name);
  console.log('args', call.function.arguments);
}
```

```ts
    },
  },

```

DATA API
Use /api/v1/data when building agents or clients that need one discoverable endpoint family for NanoGPT data tools. Direct endpoints remain available; Data API forwards requests to them and preserves request/response behavior, billing, provider options, and x402 behavior.

```bash
curl https://nano-gpt.com/api/v1/data \
  -H "Authorization: Bearer $NANOGPT_API_KEY"
```

Path-style dispatch:

```bash
curl https://nano-gpt.com/api/v1/data/web/search \
  -H "Authorization: Bearer $NANOGPT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"query":"latest AI model releases","depth":"standard"}'
```

Body-style dispatch:

```bash
curl https://nano-gpt.com/api/v1/data \
  -H "Authorization: Bearer $NANOGPT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"endpoint":"web.search","input":{"query":"latest AI model releases","depth":"standard"}}'
```

Dispatched responses expose x-nanogpt-data-endpoint and x-nanogpt-direct-endpoint. Common paths include /api/v1/data/web/search, /api/v1/data/url/scrape, /api/v1/data/firecrawl, /api/v1/data/google-maps/search, /api/v1/data/reddit, /api/v1/data/x, and /api/v1/data/hunter/domain-search.

PROMPT CACHING
NanoGPT uses implicit caching automatically on providers that support it (including OpenAI, Gemini, and many open-source provider routes).

Set top-level caching: true when the request must go to any available provider that supports prompt/input caching. This is provider capability routing, not prompt-cache annotation: it does not add Anthropic cache_control markers or configure TTLs. If no usable cache-capable provider exists for the model, the request fails. caching: true defaults to stickyprovider: true, so later matching requests from the same API key or session prefer the same provider when still usable. Use stickyprovider: false to require a cache-capable provider without stickiness. stickyProvider is also accepted as a camelCase alias.

Use prompt_caching / promptCaching / cache_control (body-level alias) when you need explicit Claude cache controls.

```ts
await client.chat.completions.create({
  model: 'model-id',
  caching: true,
  stickyprovider: false,
  messages: [{ role: 'user', content: 'Hello' }],
});
```

```ts
await client.chat.completions.create({
  model: 'anthropic/claude-sonnet-4.5',
  messages: [
    { role: 'system', content: 'Always respond with JSON.' },
    { role: 'user', content: 'Cache this conversation scaffold for 5 minutes.' },
  ],
  promptCaching: {
    enabled: true,
    ttl: '5m',
    cutAfterMessageIndex: 0,
  },
});
```

AUDIO SPEECH
POST /v1/audio/speech mirrors OpenAI’s text-to-speech API.
Default behavior is unchanged: omit `stream` (or set `false`) to receive one buffered audio file.
Set `stream: true` to receive chunked audio bytes progressively for lower time-to-first-byte (TTFB).

Streaming currently works on:
- OpenAI: `tts-1`, `tts-1-hd`, `gpt-4o-mini-tts`
- ElevenLabs via FAL: `Elevenlabs-Turbo-V2.5`, `Elevenlabs-V3`
Other TTS models ignore `stream` and return buffered audio.

```ts
// Non-streaming (default)
import { writeFile } from 'node:fs/promises';

const speech = await client.audio.speech.create({
  model: 'gpt-4o-mini-tts',
  voice: 'alloy',
  input: 'Welcome to the NanoGPT API.',
  response_format: 'mp3',
});

await writeFile('welcome.mp3', Buffer.from(await speech.arrayBuffer()));
```

```ts
// Streaming (chunked transfer encoding)
const streamingSpeech = await client.audio.speech.create({
  model: 'gpt-4o-mini-tts',
  voice: 'alloy',
  input: 'Welcome to the NanoGPT API.',
  response_format: 'mp3',
  stream: true,
});

const reader = streamingSpeech.body?.getReader();
if (!reader) throw new Error('Missing response body stream');

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  // value is a Uint8Array audio chunk
}
```

PRICING AND BILLING
- `usage` appears on non-stream JSON responses and on the final streaming chunk when `stream_options.include_usage` is enabled (or forced by helpers like prompt caching).
- Supported `usage` fields are provider-dependent and may include:
  - `prompt_tokens`
  - `completion_tokens`
  - `total_tokens`
  - `prompt_tokens_details.cached_tokens`
  - `prompt_tokens_details.audio_tokens`
  - `completion_tokens_details.reasoning_tokens`
  - `completion_tokens_details.audio_tokens`
  - `completion_tokens_details.accepted_prediction_tokens`
  - `completion_tokens_details.rejected_prediction_tokens`
  - `reasoning_tokens`
  - `citation_tokens`
  - `num_search_queries`
  - `cache_creation_input_tokens`
  - `cache_read_input_tokens`
  - `input_tokens`
- Streaming consistency note: when provider-reported usage is missing or zero, server-side normalization may backfill final usage from `x_nanogpt_pricing.inputTokens` and `x_nanogpt_pricing.outputTokens`.
- `x_nanogpt_pricing` is an extension object (final chunk only in streaming; top-level on non-stream JSON):
  - Stable/core fields:
    - `amount?: number`
    - `currency?: string`
    - `error?: { status?: number; message: string }` (`message` is sanitized)
  - Common optional fields:
    - `cost?: number`
    - `paymentSource?: string`
    - `inputTokens?: number`
    - `outputTokens?: number`
    - `cacheCost?: number`
    - `billedToTeam?: boolean`
    - `billedTeamId?: number | null`
    - `billedTeamName?: string | null`
- Compatibility rule: clients must tolerate additional fields in both `usage` and `x_nanogpt_pricing`; unknown keys should be ignored, not treated as schema errors.
- Conversation-level transactions are billed in USD with optional auto-conversion to Nano when applicable.
- Usage API: call GET https://nano-gpt.com/api/v1/usage with Authorization: Bearer $NANOGPT_API_KEY to retrieve aggregate usage for the current API key. Without dates it returns the last 30 UTC days grouped by day and model. Use from/to as YYYY-MM-DD UTC dates; both are required together, to is inclusive, to cannot be in the future, and ranges are capped at 366 days. group_by accepts day, model, day,model, and model,day (alias). scope defaults to current_key; api_key is an alias for the authenticated key, and api_key_id can only reference the current key.
- Usage API response fields: totals plus optional byDay, byModel, and byDayModel arrays contain requests, costUsd, refundedUsd, netCostUsd, inputTokens, outputTokens, reasoningTokens, and totalTokens. totalTokens is inputTokens + outputTokens; reasoningTokens are reported separately. source.rollupDays, source.liveDays, and source.missingRollupDays indicate whether results came from rollups or live aggregation. netCostUsd is floored per bucket, so row sums can differ slightly from totals when refunds cross buckets.
- Service tiers: pass service_tier: "flex" for lower-cost variable-capacity processing or service_tier: "priority" for higher-cost priority processing on /v1/chat/completions, /v1/responses, or /v1/messages. Availability is model- and provider-specific; routing overrides like X-Provider or provider-native web search affect tier pricing, and high-context pricing such as es2k can apply for GPT-5.5/GPT-5.4 where available.
- Use 429 handling with exponential backoff; subscription-free buckets allow roughly 60 RPM, while premium models may enforce tighter limits.

ERROR HANDLING AND OBSERVABILITY
- Error responses follow OpenAI envelopes: { error: { message, type, code, param } } with x-request-id headers for support.
- Authentication failures increment an abuse counter; repeated failures yield 429 with rate_limit_exceeded.
- Streaming parsers must handle terminal metadata fields (`usage`, `x_nanogpt_pricing`) and [DONE] closure explicitly.
- Log request ids alongside model and payment_source for easier reconciliation with the NanoGPT dashboard.
